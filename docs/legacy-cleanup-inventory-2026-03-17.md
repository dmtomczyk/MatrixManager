# Legacy Cleanup Inventory — 2026-03-17

This document inventories the remaining legacy Python/FastAPI surface in MatrixManager now that the TypeScript stack is the default development path.

## Summary

Current repo posture:
- **Primary stack:** TypeScript API + React/Vite UI
- **Legacy stack:** Python/FastAPI + SQLModel, retained temporarily for comparison, fallback, and installer/runtime continuity

## Bucket 1 — Safe to delete now

These are either generated artifacts or legacy UI remnants that are no longer part of the active TS path.

### Generated Python cache files
- `app/__pycache__/__init__.cpython-313.pyc`
- `app/__pycache__/main.cpython-313.pyc`
- `tests/__pycache__/conftest.cpython-313-pytest-8.3.5.pyc`
- `tests/__pycache__/test_api_requirements.cpython-313-pytest-8.3.5.pyc`
- `tests/__pycache__/test_ui_surfaces.cpython-313-pytest-8.3.5.pyc`

Why safe:
- generated bytecode only
- should not be committed or relied on

### Legacy static HTML shells
- `app/static/canvas.html`
- `app/static/home.html`

Why safe:
- current app routing is served by TS/server-rendered pages and React shell paths
- these legacy standalone HTML pages are no longer in the active route flow

### Legacy backend implementation

- still referenced by explicit legacy scripts

### Legacy/backend dependency documentation

Why keep for now:

## Bucket 3 — Migrate first before deletion

These should not be deleted until replacement work is done.

### Python test suite
- `tests/conftest.py`
- `tests/test_api_requirements.py`
- `tests/test_ui_surfaces.py`
- any other Python tests under `tests/`

Why not yet:
- current automated test coverage still references FastAPI/SQLModel test client patterns
- TS-native API/page coverage should exist before removing them

Recommended replacement:
- Playwright coverage for end-to-end TS flows
- TS/API route tests around auth, CRUD, audit, runtime, and persistence flows

### Package scripts still calling Python utilities
Current package.json still includes:
- `db:reset` → `scripts/reset_db.py`
- `db:seed` → `scripts/seed_sample_data.py`
- `dev:api:legacy` → uvicorn app.main:app

Why not yet:
- they are explicit escape hatches/reference paths during staged cleanup
- should be removed only after TS-native equivalents or a clear deprecation decision

Recommended replacement:
- `db:reset:ts`
- `db:seed:ts`
- eventual removal of `dev:legacy` once confidence is high

### Installer/runtime production path
Potentially Python-bound today:
- `Dockerfile`
- `docker-compose.yml`
- shell lifecycle scripts
- `.env.example` fields that still reflect Python/runtime assumptions

Why not yet:
- needs a deliberate TS production/runtime plan
- should be migrated as a coherent deployment transition, not piecemeal

## Suggested deletion order

### Next low-risk deletion pass
1. remove Python cache files
2. remove `app/static/canvas.html`
3. remove `app/static/home.html`
4. update `.gitignore` if needed to keep cache artifacts out permanently

### Next migration pass before broader deletion
1. add TS-native reset/seed commands
2. add TS-native automated coverage for critical flows
3. decide TS production runtime path for Docker/install scripts

### After that
1. remove Python test harness
2. remove Python helper scripts
3. remove `dev:legacy`
4. remove `requirements.txt`
5. remove `app/main.py` and remaining legacy backend package files

## Recommendation

Proceed with two things in parallel:

1. low-risk deletion of generated Python cache artifacts
2. TS-first replacement work for runtime packaging and remaining legacy tests

Do **not** remove the legacy backend implementation, Python utility scripts, or installer/runtime materials until:
- TS-native reset/seed exists
- TS-native test coverage is stronger
- deployment/runtime strategy is explicitly moved off the Python backend

This inventory should be kept in sync with future legacy removal passes.
